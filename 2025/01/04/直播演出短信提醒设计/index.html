直播演出短信提醒设计

我们可以考虑使用时间轮算法，或者基于消息队列的延迟消息，或者使用数据库索引和轮询分片，或者使用外部调度服务如Quartz，或者利用Redis的sorted set等。

由于问题是开放式的，需要给出设计方案，考虑高并发、高可用、可扩展性等。

我们可以提出几种方案，比较优劣，并推荐一种。

常见方案：

  1. 使用数据库定期扫描，但优化：按演出开始时间索引，只扫描即将开始的演出（比如未来10分钟内），而不是全表。但依然要扫描演出表，然后对每个演出找出订阅用户。可以加条件限制，减小扫描范围。
  2. 使用消息队列延迟消息：当用户订阅时，计算演出开始前5分钟的时间戳，发送一个延迟消息到队列，到时间后消费消息发送短信。但需要处理用户取消订阅等情况。并且如果演出时间改变，需要更新消息。
  3. 使用Redis的Sorted Set：将提醒任务按触发时间戳排序，定期从Redis中获取到期的任务进行处理。可以实现精确触发，且性能高。
  4. 使用时间轮算法（如Netty的HashedWheelTimer）或类似Quartz的调度框架。
  5. 基于数据库的分片轮询：将演出ID哈希后分布到多个队列中，每个队列独立扫描。

需要考虑用户订阅提醒的操作频率，以及演出数量的规模。

假设演出数量可能很多，用户订阅量也大。需要设计一个可水平扩展的方案。

我们可以结合消息队列和数据库：当用户订阅时，将提醒任务发送到延迟队列（如RabbitMQ的延迟插件，或者使用Kafka的时间戳索引）。但延迟消息通常有最大延迟时间限制，比如RabbitMQ延迟插件可能有限制，但可以用定时任务轮询消息队列。

另一种方案：将演出开始时间作为分片键，定期扫描未来一小段时间内的演出，然后将这些演出的提醒任务批量提交到队列或直接处理。这样扫描数据库只针对即将开始的演出，数据量小。

由于用户订阅是实时添加的，我们需要确保新订阅的用户也能收到提醒，即使订阅时间晚于提醒触发时间（比如演出开始前1分钟才订阅，那么应该立即发送？但需求是开场前5分钟提醒，如果用户订阅时距离演出开始不足5分钟，应该怎么处理？可能也需要发送，或者忽略。可以定义：如果订阅时间距离演出开始时间小于5分钟，则立即发送短信；否则，安排在5分钟前发送。但这里我们只考虑正常情况：在开场前5分钟发送。

设计一个健壮的系统需要考虑：

  • 用户订阅提醒：存储用户ID和演出ID，关联演出开始时间。
  • 当演出时间变更时，需要更新提醒任务。
  • 取消订阅：需要取消未发送的提醒。

我们可以使用消息队列延迟消息，但是消息队列的延迟消息通常不支持动态取消（需要额外处理）。使用Redis sorted set可以方便地移除。

一个常用模式：使用Redis的ZSet存储待发送提醒，score为触发时间戳（演出开始时间-5分钟）。定期从Redis中获取当前时间戳之前的任务，发送短信，然后从ZSet中删除。同时，为了防止丢失，可以将任务持久化到数据库，Redis作为实时调度层，数据库作为备份和恢复。

具体流程：

  • 用户订阅提醒时，计算触发时间戳t = 演出开始时间 - 5分钟。
  • 将提醒任务信息（如用户ID、演出ID、手机号等）存入数据库（提醒任务表）。
  • 同时，将任务以t为score存入Redis ZSet（key可能是“reminder_tasks”），value可以是一个唯一ID（比如任务ID），或者序列化信息。为了支持取消，还需要一个映射存储任务详情。
  • 启动一个调度服务，不断从Redis ZSet中获取score <= 当前时间戳的任务（使用ZRANGEBYSCORE），每批取一定数量，然后处理发送短信。处理完成后从ZSet中移除这些任务，并更新数据库状态为已发送。
  • 为了容错，如果Redis宕机，可以从数据库恢复未发送的任务（通过状态和触发时间）重新加载到Redis。可以定期将未来一定时间内的任务加载到Redis。

这个方案结合了Redis的高性能和数据库的持久化，是比较通用的做法。

另外，为了避免全量扫描数据库，我们可以对数据库的提醒任务表建立索引（演出开始时间、状态）。但即便如此，如果数据量大，频繁扫描未来5分钟的任务也可能有压力。使用Redis作为缓冲区可以减少数据库扫描。

另一种思路：使用时间轮算法，但需要自己实现，且不便于持久化。

考虑到实际业务，可能使用成熟的调度框架如Quartz，但Quartz也是基于数据库扫描，会有锁竞争。

因此，推荐使用Redis ZSet + 数据库的方案。

还需要考虑分布式环境下的并发处理，多个消费者从Redis取任务时，要保证任务不被重复消费。可以使用Redis的ZPOPMIN或Lua脚本实现原子性取任务。但ZPOPMIN只支持单个，可以批量取并使用Lua脚本或事务。

另外，要处理任务失败重试。

还需要考虑演出时间变更：如果演出时间修改，需要更新提醒任务。我们可以先取消原来的任务（从Redis删除，更新数据库状态），再重新插入新任务。取消时需要知道任务ID。

用户取消订阅类似。

总之，这个方案可以满足需求。

现在，根据问题，给出一个清晰的设计思路，并解释为什么避免全量扫描。

针对你的业务场景（直播演出开场前5分钟短信提醒，用户动态订阅，避免全量扫描数据库），我设计一个基于 Redis ZSet + 数据库持久化 的高效解决方案。该方案既能支持大量用户和演出数据，又能实时响应新订阅，同时避免频繁的全表扫描。

核心思路

  • 触发时间计算：每个提醒任务的触发时间为 演出开始时间 - 5分钟。
  • 实时调度层：使用 Redis 的有序集合（ZSet）存储待触发的任务，以触发时间戳作为分数（score），支持毫秒级范围查询和弹出。
  • 持久化层：使用数据库（如 MySQL）记录所有提醒任务的详细信息，包括用户、演出、状态等，用于数据备份、恢复和变更管理。
  • 消费者进程：多个 Worker 不断从 Redis 中拉取到期的任务，发送短信，并更新数据库状态，确保任务被可靠处理。

详细设计

1. 数据表结构

提醒任务表（reminder_task）

字段名 类型 说明
id bigint 主键，唯一标识
user_id varchar 用户ID
show_id varchar 演出ID
phone varchar 手机号
show_time datetime 演出开始时间
trigger_time datetime 应触发时间（show_time - 5分钟）
status tinyint 状态：0-待触发，1-已触发，2-已取消
retry_count tinyint 失败重试次数
create_time datetime 创建时间
update_time datetime 最后更新时间

索引建议：(trigger_time, status) 用于恢复未完成任务；(show_id, user_id) 用于去重。

2. 订阅流程（用户添加提醒）

当用户订阅某个演出的提醒时：

  1. 计算 trigger_time = show_time - 5分钟。
  2. 插入一条记录到 reminder_task 表，状态为 待触发。
  3. 将任务信息以 JSON 或唯一ID的形式存入 Redis：
    • 使用一个固定的 ZSet Key，例如 reminder:zset。
    • 成员（member）：可以是任务ID，或者包含用户ID、演出ID的字符串（推荐用任务ID，便于取消）。
    • 分数（score）：trigger_time 的时间戳（毫秒或秒）。
  4. 如果此时 trigger_time 已经小于当前时间（即用户订阅时已过提醒点），可直接触发发送，或仍按正常流程等待下一次扫描（取决于业务规则，通常应立即发送）。

3. 提醒触发流程（消费者 Worker）

  • 启动多个 Worker 进程（可水平扩展），每个 Worker 定时（如每秒）执行以下操作：
    1. 使用 Redis 的 ZRANGEBYSCORE 命令获取当前时间戳之前的所有任务（score <= now），并限制每次取出的数量（例如 100 条），防止单次处理过多。
    2. 为了原子性，可使用 Lua 脚本：先查询出这批任务，然后从 ZSet 中移除它们（ZREMRANGEBYSCORE）。
    3. 对取出的每个任务：
      • 解析出任务ID，查询数据库获取完整信息（或直接从Redis member中携带必要信息，如用户手机号）。
      • 调用短信发送接口。
      • 根据发送结果更新数据库状态：成功则置为 已触发；失败则记录重试次数，若未超限则重新计算下次重试时间并重新入 Redis（或由死信队列处理）。
    4. 如果发送过程中 Worker 崩溃，未确认的任务会留在 Redis（已移除）？实际上，由于我们已经从 Redis 移除了任务，失败的任务需要依赖数据库状态进行恢复。可引入“处理中”状态，但为简化，可以先从数据库加载任务并更新状态，失败后重新插入 Redis（利用重试机制）。更健壮的做法是使用消息队列的 ACK 机制，但 Redis 方案也能通过记录处理日志来保证最终一致性。

4. 演出时间变更处理

如果演出时间修改：

  1. 找到该演出关联的所有 待触发 提醒任务。
  2. 将这些任务从 Redis ZSet 中删除（可根据任务ID删除，需维护一个从 member 到 ID 的映射，或直接用任务ID作为 member）。
  3. 更新数据库中的 show_time 和 trigger_time，状态保持不变。
  4. 重新将任务插入 Redis ZSet（新的触发时间）。

5. 用户取消订阅

  • 更新数据库状态为 已取消。
  • 同时从 Redis ZSet 中删除对应任务（通过任务ID）。

6. 系统启动/故障恢复

  • 系统启动时，从数据库中查询 trigger_time 在 [当前时间, 当前时间 + 预加载窗口] 范围内且状态为 待触发 的任务，批量加载到 Redis ZSet 中。预加载窗口可设为未来 1 小时，避免一次性加载过多。
  • 定期扫描数据库，将未来更长时间的任务逐步预热到 Redis。

为什么避免全量扫描？

  • 数据库只作为持久化存储，日常提醒触发完全依赖 Redis 的实时扫描，数据库仅用于查询特定任务（如恢复、变更），不需要每隔 5 分钟全表扫描。
  • Redis ZSet 的 ZRANGEBYSCORE 操作时间复杂度为 O(log(N)+M)，性能极高，即使有百万级待触发任务也能毫秒级返回。
  • 通过分页取出和批量处理，避免了单次处理过多任务导致阻塞。

扩展性考虑

  • Redis 分片：如果单 Redis 实例压力过大，可按演出ID哈希分片，每个分片对应一个 ZSet，Worker 分别消费。
  • 高可用：Redis 采用哨兵或集群模式；数据库主从备份。
  • 短信发送限流：控制 Worker 消费速率，对接短信网关的 QPS 限制。
  • 监控告警：监控 Redis ZSet 积压量、发送成功率、失败重试次数。

备选方案对比

方案 优点 缺点
数据库轮询 + 索引 简单，不引入新组件 随数据量增大性能下降，频繁扫描有压力
消息队列延迟消息 精确触发，消息持久化 不支持动态取消/修改，延迟时间有限制
Quartz 成熟，支持集群 基于数据库锁，性能瓶颈，不适合海量任务
Redis ZSet 高性能，支持动态增删，易水平扩展 需配合数据库做持久化，需处理恢复逻辑

综合来看，Redis ZSet + 数据库 最适合你的场景，兼顾了性能、灵活性和数据可靠性。

总结

该设计方案利用 Redis 的有序集合作为实时调度器，避免了全量扫描数据库；数据库作为可靠存储，处理变更和恢复。通过合理的任务加载和消费者并发控制，可以支撑大规模用户订阅和演出提醒，满足业务需求。
