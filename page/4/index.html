直播演出短信提醒设计

我们可以考虑使用时间轮算法，或者基于消息队列的延迟消息，或者使用数据库索引和轮询分片，或者使用外部调度服务如Quartz，或者利用Redis的sorted set等。

由于问题是开放式的，需要给出设计方案，考虑高并发、高可用、可扩展性等。

我们可以提出几种方案，比较优劣，并推荐一种。

常见方案：

  1. 使用数据库定期扫描，但优化：按演出开始时间索引，只扫描即将开始的演出（比如未来10分钟内），而不是全表。但依然要扫描演出表，然后对每个演出找出订阅用户。可以加条件限制，减小扫描范围。
  2. 使用消息队列延迟消息：当用户订阅时，计算演出开始前5分钟的时间戳，发送一个延迟消息到队列，到时间后消费消息发送短信。但需要处理用户取消订阅等情况。并且如果演出时间改变，需要更新消息。
  3. 使用Redis的Sorted Set：将提醒任务按触发时间戳排序，定期从Redis中获取到期的任务进行处理。可以实现精确触发，且性能高。
  4. 使用时间轮算法（如Netty的HashedWheelTimer）或类似Quartz的调度框架。
  5. 基于数据库的分片轮询：将演出ID哈希后分布到多个队列中，每个队列独立扫描。

需要考虑用户订阅提醒的操作频率，以及演出数量的规模。

假设演出数量可能很多，用户订阅量也大。需要设计一个可水平扩展的方案。

我们可以结合消息队列和数据库：当用户订阅时，将提醒任务发送到延迟队列（如RabbitMQ的延迟插件，或者使用Kafka的时间戳索引）。但延迟消息通常有最大延迟时间限制，比如RabbitMQ延迟插件可能有限制，但可以用定时任务轮询消息队列。

另一种方案：将演出开始时间作为分片键，定期扫描未来一小段时间内的演出，然后将这些演出的提醒任务批量提交到队列或直接处理。这样扫描数据库只针对即将开始的演出，数据量小。

由于用户订阅是实时添加的，我们需要确保新订阅的用户也能收到提醒，即使订阅时间晚于提醒触发时间（比如演出开始前1分钟才订阅，那么应该立即发送？但需求是开场前5分钟提醒，如果用户订阅时距离演出开始不足5分钟，应该怎么处理？可能也需要发送，或者忽略。可以定义：如果订阅时间距离演出开始时间小于5分钟，则立即发送短信；否则，安排在5分钟前发送。但这里我们只考虑正常情况：在开场前5分钟发送。

设计一个健壮的系统需要考虑：

  • 用户订阅提醒：存储用户ID和演出ID，关联演出开始时间。
  • 当演出时间变更时，需要更新提醒任务。
  • 取消订阅：需要取消未发送的提醒。

我们可以使用消息队列延迟消息，但是消息队列的延迟消息通常不支持动态取消（需要额外处理）。使用Redis sorted set可以方便地移除。

一个常用模式：使用Redis的ZSet存储待发送提醒，score为触发时间戳（演出开始时间-5分钟）。定期从Redis中获取当前时间戳之前的任务，发送短信，然后从ZSet中删除。同时，为了防止丢失，可以将任务持久化到数据库，Redis作为实时调度层，数据库作为备份和恢复。

具体流程：

  • 用户订阅提醒时，计算触发时间戳t = 演出开始时间 - 5分钟。
  • 将提醒任务信息（如用户ID、演出ID、手机号等）存入数据库（提醒任务表）。
  • 同时，将任务以t为score存入Redis ZSet（key可能是“reminder_tasks”），value可以是一个唯一ID（比如任务ID），或者序列化信息。为了支持取消，还需要一个映射存储任务详情。
  • 启动一个调度服务，不断从Redis ZSet中获取score <= 当前时间戳的任务（使用ZRANGEBYSCORE），每批取一定数量，然后处理发送短信。处理完成后从ZSet中移除这些任务，并更新数据库状态为已发送。
  • 为了容错，如果Redis宕机，可以从数据库恢复未发送的任务（通过状态和触发时间）重新加载到Redis。可以定期将未来一定时间内的任务加载到Redis。

这个方案结合了Redis的高性能和数据库的持久化，是比较通用的做法。

另外，为了避免全量扫描数据库，我们可以对数据库的提醒任务表建立索引（演出开始时间、状态）。但即便如此，如果数据量大，频繁扫描未来5分钟的任务也可能有压力。使用Redis作为缓冲区可以减少数据库扫描。

另一种思路：使用时间轮算法，但需要自己实现，且不便于持久化。

考虑到实际业务，可能使用成熟的调度框架如Quartz，但Quartz也是基于数据库扫描，会有锁竞争。

因此，推荐使用Redis ZSet + 数据库的方案。

还需要考虑分布式环境下的并发处理，多个消费者从Redis取任务时，要保证任务不被重复消费。可以使用Redis的ZPOPMIN或Lua脚本实现原子性取任务。但ZPOPMIN只支持单个，可以批量取并使用Lua脚本或事务。

另外，要处理任务失败重试。

还需要考虑演出时间变更：如果演出时间修改，需要更新提醒任务。我们可以先取消原来的任务（从Redis删除，更新数据库状态），再重新插入新任务。取消时需要知道任务ID。

用户取消订阅类似。

总之，这个方案可以满足需求。

现在，根据问题，给出一个清晰的设计思路，并解释为什么避免全量扫描。

针对你的业务场景（直播演出开场前5分钟短信提醒，用户动态订阅，避免全量扫描数据库），我设计一个基于 Redis ZSet + 数据库持久化 的高效解决方案。该方案既能支持大量用户和演出数据，又能实时响应新订阅，同时避免频繁的全表扫描。

核心思路

  • 触发时间计算：每个提醒任务的触发时间为 演出开始时间 - 5分钟。
  • 实时调度层：使用 Redis 的有序集合（ZSet）存储待触发的任务，以触发时间戳作为分数（score），支持毫秒级范围查询和弹出。
  • 持久化层：使用数据库（如 MySQL）记录所有提醒任务的详细信息，包括用户、演出、状态等，用于数据备份、恢复和变更管理。
  • 消费者进程：多个 Worker 不断从 Redis 中拉取到期的任务，发送短信，并更新数据库状态，确保任务被可靠处理。

详细设计

1. 数据表结构

提醒任务表（reminder_task）

字段名 类型 说明
id bigint 主键，唯一标识
user_id varchar 用户ID
show_id varchar 演出ID
phone varchar 手机号
show_time datetime 演出开始时间
trigger_time datetime 应触发时间（show_time - 5分钟）
status tinyint 状态：0-待触发，1-已触发，2-已取消
retry_count tinyint 失败重试次数
create_time datetime 创建时间
update_time datetime 最后更新时间

索引建议：(trigger_time, status) 用于恢复未完成任务；(show_id, user_id) 用于去重。

2. 订阅流程（用户添加提醒）

当用户订阅某个演出的提醒时：

  1. 计算 trigger_time = show_time - 5分钟。
  2. 插入一条记录到 reminder_task 表，状态为 待触发。
  3. 将任务信息以 JSON 或唯一ID的形式存入 Redis：
    • 使用一个固定的 ZSet Key，例如 reminder:zset。
    • 成员（member）：可以是任务ID，或者包含用户ID、演出ID的字符串（推荐用任务ID，便于取消）。
    • 分数（score）：trigger_time 的时间戳（毫秒或秒）。
  4. 如果此时 trigger_time 已经小于当前时间（即用户订阅时已过提醒点），可直接触发发送，或仍按正常流程等待下一次扫描（取决于业务规则，通常应立即发送）。

3. 提醒触发流程（消费者 Worker）

  • 启动多个 Worker 进程（可水平扩展），每个 Worker 定时（如每秒）执行以下操作：
    1. 使用 Redis 的 ZRANGEBYSCORE 命令获取当前时间戳之前的所有任务（score <= now），并限制每次取出的数量（例如 100 条），防止单次处理过多。
    2. 为了原子性，可使用 Lua 脚本：先查询出这批任务，然后从 ZSet 中移除它们（ZREMRANGEBYSCORE）。
    3. 对取出的每个任务：
      • 解析出任务ID，查询数据库获取完整信息（或直接从Redis member中携带必要信息，如用户手机号）。
      • 调用短信发送接口。
      • 根据发送结果更新数据库状态：成功则置为 已触发；失败则记录重试次数，若未超限则重新计算下次重试时间并重新入 Redis（或由死信队列处理）。
    4. 如果发送过程中 Worker 崩溃，未确认的任务会留在 Redis（已移除）？实际上，由于我们已经从 Redis 移除了任务，失败的任务需要依赖数据库状态进行恢复。可引入“处理中”状态，但为简化，可以先从数据库加载任务并更新状态，失败后重新插入 Redis（利用重试机制）。更健壮的做法是使用消息队列的 ACK 机制，但 Redis 方案也能通过记录处理日志来保证最终一致性。

4. 演出时间变更处理

如果演出时间修改：

  1. 找到该演出关联的所有 待触发 提醒任务。
  2. 将这些任务从 Redis ZSet 中删除（可根据任务ID删除，需维护一个从 member 到 ID 的映射，或直接用任务ID作为 member）。
  3. 更新数据库中的 show_time 和 trigger_time，状态保持不变。
  4. 重新将任务插入 Redis ZSet（新的触发时间）。

5. 用户取消订阅

  • 更新数据库状态为 已取消。
  • 同时从 Redis ZSet 中删除对应任务（通过任务ID）。

6. 系统启动/故障恢复

  • 系统启动时，从数据库中查询 trigger_time 在 [当前时间, 当前时间 + 预加载窗口] 范围内且状态为 待触发 的任务，批量加载到 Redis ZSet 中。预加载窗口可设为未来 1 小时，避免一次性加载过多。
  • 定期扫描数据库，将未来更长时间的任务逐步预热到 Redis。

为什么避免全量扫描？

  • 数据库只作为持久化存储，日常提醒触发完全依赖 Redis 的实时扫描，数据库仅用于查询特定任务（如恢复、变更），不需要每隔 5 分钟全表扫描。
  • Redis ZSet 的 ZRANGEBYSCORE 操作时间复杂度为 O(log(N)+M)，性能极高，即使有百万级待触发任务也能毫秒级返回。
  • 通过分页取出和批量处理，避免了单次处理过多任务导致阻塞。

扩展性考虑

  • Redis 分片：如果单 Redis 实例压力过大，可按演出ID哈希分片，每个分片对应一个 ZSet，Worker 分别消费。
  • 高可用：Redis 采用哨兵或集群模式；数据库主从备份。
  • 短信发送限流：控制 Worker 消费速率，对接短信网关的 QPS 限制。
  • 监控告警：监控 Redis ZSet 积压量、发送成功率、失败重试次数。

备选方案对比

方案 优点 缺点
数据库轮询 + 索引 简单，不引入新组件 随数据量增大性能下降，频繁扫描有压力
消息队列延迟消息 精确触发，消息持久化 不支持动态取消/修改，延迟时间有限制
Quartz 成熟，支持集群 基于数据库锁，性能瓶颈，不适合海量任务
Redis ZSet 高性能，支持动态增删，易水平扩展 需配合数据库做持久化，需处理恢复逻辑

综合来看，Redis ZSet + 数据库 最适合你的场景，兼顾了性能、灵活性和数据可靠性。

总结

该设计方案利用 Redis 的有序集合作为实时调度器，避免了全量扫描数据库；数据库作为可靠存储，处理变更和恢复。通过合理的任务加载和消费者并发控制，可以支撑大规模用户订阅和演出提醒，满足业务需求。

线上Redis宕机优化策略

思考过程（Thinking）：

嗯，用户问的是线上Redis机器宕机后如何优化。这是一个非常典型的运维和架构问题，用户的核心需求是知道在发生宕机后，如何进行系统性的改进和优化，以避免未来再次发生类似问题或减少影响。

问题本身有一定复杂度，需要从多个层面考虑。用户可能不仅想知道应急处理，更想知道长期的根本解决方案。我应该提供一个结构清晰的优化思路，涵盖从快速恢复、高可用架构、数据持久化到日常运维等多个方面。

可以从几个关键维度来组织回答：首先是高可用架构的改进，避免单点故障；其次是数据持久化和恢复策略，确保数据不丢；然后是日常的监控、容量规划和故障演练，做到提前预防。最后可以提一下关键业务的数据零丢失方案，作为更高级的优化方向。这样回答既全面又有层次，能覆盖用户的实际需求。

线上Redis宕机确实让人紧张。优化可以从事前防御和事后恢复两个维度系统性地进行：

1. 高可用架构（防止单点故障）

这是避免因宕机导致业务中断的核心。

  • 主从复制 + 哨兵：为每个Master配一个Slave，再用至少3个哨兵节点监控，实现主节点故障的自动摘除与选举。
  • Redis Cluster：若数据量或QPS很大，建议直接使用。通过数据分片将压力分散，节点宕机只影响部分Slot，且内部支持自动故障转移。

2. 数据持久化（快速恢复）

如果宕机后重启，持久化机制决定了恢复速度和数据完整性。

  • AOF + rewrite：建议开启AOF，并设置appendfsync everysec（最多丢1秒数据）。配合定时rewrite，可兼顾数据安全和恢复速度。
  • 混合持久化：Redis 4.0以上版本推荐开启（aof-use-rdb-preamble yes）。用RDB做全量快照，用AOF存增量，重启加载速度极快。

3. 事前防御与容量规划

  • 合理设置maxmemory：避免内存写满导致OOM或被系统kill。
  • 慢查询优化：排查并优化慢查询命令，防止Redis阻塞。
  • 禁用高风险命令：如 Keys、Flushall 等，防止误操作导致不可用。

4. 监控与快速定位

  • 监控关键指标：如命中率、QPS、慢查询数。
  • 设置分级告警：如CPU超过80%触发警告，连接数突降到0则触发电话告警，便于快速响应。

5. 降级与熔断机制（业务侧防护）

如果Redis宕机，业务层要有预案防止雪崩：

  • 本地缓存兜底：如Caffeine或Guava Cache，Redis挂了可以先读本地。
  • 空结果缓存：防止缓存穿透打垮数据库。
  • 熔断返回默认值：直接返回默认数据（如默认商品信息），保证主流程跑通。

如果你们是单节点，当务之急是先搭建主从哨兵架构；如果是集群，重点应放在分析内存使用和优化持久化配置上。

GraphQL解决痛点与引入原因

思考过程（Thinking）：

我们被问到：”graphql能解决什么痛点，为什么要引入”。这是一个开放性问题，需要解释GraphQL解决了哪些痛点，以及为什么应该引入它。我们需要给出一个清晰、有结构的回答。

考虑回答要点：

  • 传统REST API的痛点：过度获取（over-fetching）和不足获取（under-fetching）、多个端点、版本控制、弱类型、多次往返等。
  • GraphQL的优势：精确获取所需数据、单一端点、强类型Schema、声明式数据获取、聚合多个资源、减少网络请求、更好的开发体验（自省、工具支持）。
  • 引入GraphQL的原因：提高前端开发效率、减少后端变更的影响、更好的性能、团队协作等。

回答应该简明扼要，但要覆盖关键点。可能还需要提到GraphQL不是银弹，有学习曲线和复杂性。但问题主要是问解决什么痛点，为什么引入，所以可以侧重于好处。

我将用中文回答。

GraphQL 主要解决了传统 RESTful API 在使用过程中常见的几大痛点：

1. 过度获取（Over-fetching）与不足获取（Under-fetching）

  • REST 的痛点：客户端往往只能调用固定结构的接口，返回的数据可能包含大量不需要的字段（过度获取）；或者需要的数据分散在多个接口中，不得不发起多次请求拼凑（不足获取）。
  • GraphQL 的解决：客户端可以精确声明所需字段，服务器只返回这些字段，既避免浪费流量，又保证不多次请求。

2. 多端点混乱与版本管理

  • REST 的痛点：随着业务发展，接口数量激增，维护多个端点（如 /users, /users/{id}/posts）成本高，且修改接口往往需要新建版本（如 /v2/users），导致代码膨胀。
  • GraphQL 的解决：单一端点（通常是 /graphql），所有查询都通过该端点，Schema 内可自由扩展字段，无需显式版本控制（通过弃用字段或添加新字段平滑升级）。

3. 弱类型与文档维护困难

  • REST 的痛点：多数 REST API 缺乏强类型契约，文档常与实现脱节，需要手动维护（如 Swagger 虽能生成，但依赖注解）。
  • GraphQL 的解决：强类型 Schema 自描述，既是契约又是文档，配合 GraphiQL 等工具可实时探索和调试。类型系统让前后端类型安全成为可能。

4. 多次往返与性能问题

  • REST 的痛点：复杂页面常常需要串行或并行调用多个接口，导致网络延迟累积，尤其在移动端更为明显。
  • GraphQL 的解决：一次请求获取所有需要的数据，减少网络往返次数。同时，解析器层面可以优化数据加载（如使用 DataLoader 避免 N+1 查询）。

5. 前端依赖后端数据结构

  • REST 的痛点：前端 UI 需要的数据结构往往与后端返回的结构不一致，前端需做大量数据转换，且后端变更可能直接影响前端。
  • GraphQL 的解决：前端可按需重组数据，查询结构可直接对应 UI 组件需求，降低前后端耦合，让前端更自主。

为什么要引入 GraphQL？

引入 GraphQL 的核心动机是提升开发效率与用户体验：

  • 快速迭代：前端无需等待后端为每个新需求新建接口，可自行扩展查询。
  • 性能优化：减少传输数据量和请求次数，提升应用响应速度。
  • 团队自治：前后端可并行开发，基于 Schema 协商，减少沟通成本。
  • 工具生态：丰富的工具（如 Apollo、Relay、GraphQL Code Generator）提供缓存、类型生成、订阅等能力，进一步简化开发。

当然，GraphQL 也带来查询复杂度控制、缓存策略变化等新挑战，但它在数据聚合层、BFF（Backend for Frontend）场景下表现卓越，尤其适合中大型项目或需要快速迭代的产品。

Nacos配置热更新实现方式解析

Nacos作为配置中心，其核心优势就在于支持配置的热更新。当Nacos上的配置文件发生变化后，对应的Spring项目通常不需要重启即可使新配置生效。

实现热更新的关键在于正确的代码配置。下面为你详细解析两种主流的实现方式。

🔧 两种热更新实现方式

你可以根据项目结构和个人偏好选择以下任意一种方法。

特性 @ConfigurationProperties 方式 @RefreshScope + @Value 方式
推荐度 推荐 可用，但部分版本可能不生效
原理 将配置属性绑定到一个JavaBean上，Spring会自动刷新该Bean的属性值。 标记一个Bean为“刷新作用域”，配置变更时会重建该Bean。
优点 - 代码整洁：集中管理配置。 - 无需额外注解：本身支持热更新。 - 类型安全 - 灵活：可加在单个需要更新的类上。
缺点 需要创建一个配置类。 - 分散：配置项多时注解分散。 - 潜在不稳定

1. 使用 @ConfigurationProperties 注解（推荐）

这种方式是最优雅和推荐的做法。它通过创建一个配置属性类来集中映射配置文件中的键值对。

实现步骤：

  1. 创建配置类：定义一个类，使用 @Component 将其纳入Spring容器管理，并使用 @ConfigurationProperties(prefix = "你的前缀") 注解指定配置前缀。
  2. 注入使用：在需要用到配置的Service或Controller中，直接注入这个配置类即可。

当Nacos中的配置修改后，CartProperties 类中的字段值会被Spring Boot自动更新。

2. 使用 @RefreshScope 和 @Value 注解

这种方式更为直接，适用于快速对单个配置项进行热更新。

实现步骤：

  1. 注解类：在使用了 @Value 注解的类上（通常是Controller或Service），额外添加 @RefreshScope 注解。
  2. 注入属性：使用 @Value("${property.name}") 注入配置值。

当配置变更时，Nacos客户端会通知应用，@RefreshScope 注解下的Bean会被标记为脏状态，下次请求时会重新创建，从而获取到最新的配置值。

💡 重要前提与补充知识

为了确保热更新功能正常工作，以下几点需要特别注意：

  • 正确的项目配置：你的Spring Cloud项目必须已经正确集成Nacos Config。这通常意味着在 pom.xml 中引入了 spring-cloud-starter-alibaba-nacos-config 依赖，并配置了 bootstrap.yml 文件（优先级高于 application.yml）来指定Nacos服务器地址和应用名。
  • 配置的自动刷新：在Nacos中，配置的 auto-refresh 默认是开启的。但在引入扩展配置时，最好显式设置为 true。
  • 配置优先级：如果同一个属性在多个地方（如Nacos的不同配置文件、本地配置文件）都有定义，Spring Boot会遵循一定的优先级顺序。了解这一点有助于排查配置不生效的问题。
  • 处理复杂更新：对于更复杂的更新逻辑（例如配置变更后需要重建数据库连接池或线程池），可以监听Spring的 RefreshEvent 事件，在回调中编写自定义逻辑。

💎 如何选择？

  • 对于新建项目或结构规整的项目，强烈建议采用 @ConfigurationProperties 方式，它更利于维护和扩展。
  • 如果只是对现有代码的某个配置项进行快速改造，使用 @RefreshScope + @Value 更为便捷。

注解@Transactional的参数

在Spring框架中，@Transactional注解用于声明一个方法或类需要进行事务管理。它可以应用于服务层的方法，以确保数据的一致性和完整性。以下是@Transactional注解的常用参数：

常用参数

  1. propagation
    • 类型：Propagation
    • 默认值：Propagation.REQUIRED
    • 描述：定义事务的传播行为，如是否支持当前事务，是否新建事务等。
  2. isolation
    • 类型：Isolation
    • 默认值：Isolation.DEFAULT
    • 描述：定义事务的隔离级别，控制多个事务之间的相互影响程度，如 READ_COMMITTED, REPEATABLE_READ 等。
  3. timeout
    • 类型：int
    • 默认值：-1（无限制）
    • 描述：事务的超时时间，单位为秒。如果事务超过这个时间将会被回滚。
  4. readOnly
    • 类型：boolean
    • 默认值：false
    • 描述：指示事务是否为只读事务。设置为 true 可以优化性能，但并不强制限制对数据库的修改。
  5. rollbackFor
    • 类型：Class<?>[]
    • 默认值：{RuntimeException.class}
    • 描述：指定哪些异常会导致事务回滚。
  6. noRollbackFor
    • 类型：Class<?>[]
    • 默认值：{}（空数组）
    • 描述：指定哪些异常不会导致事务回滚。
  7. rollbackForClassName
    • 类型：String[]
    • 描述：与 rollbackFor 类似，但是通过异常的类名来指定。
  8. noRollbackForClassName
    • 类型：String[]
    • 描述：与 noRollbackFor 类似，通过异常的类名来指定。

消息队列故障兜底方案

1. 快速分析：MQ故障的影响

MQ故障通常导致两类问题：生产者无法发送消息，以及消费者无法拉取消息。最严重的情况是，链路阻断引发的业务逻辑失败，可能直接影响用户体验。

2. 兜底方案核心架构

建议采取本地缓冲 + 链路降级 + 数据对账的组合策略。

  • 短期策略（故障发生0-5分钟内）：业务熔断与降级
    • 同步转异步降级：核心业务（如订单支付成功）若MQ不可用，应在代码中捕获异常后，将待处理任务写入数据库本地事务表，并立即返回成功。随后由定时任务扫表，尝试重新投递。
    • 功能降级：非核心业务（如发送积分、日志记录）可直接丢弃或打印告警后忽略。例如，用户浏览商品日志丢失，不影响交易。
  • 中期策略（故障发生5-30分钟内）：本地文件缓冲（高可靠场景）
    • 实现原理：当MQ客户端连续重试失败，触发告警后，系统将消息体序列化后，追加写入本地的日志文件（如{timestamp}.bak）。同时，启动一个守护线程监控MQ恢复状况。一旦MQ恢复，立即按顺序读取本地文件，将积压消息重新发送。
    • 注意：需考虑磁盘空间占用，并做好文件滚动（如每小时新建一个文件，自动删除7天前的文件）。
  • 长期策略（根本治理）：多链路与对账
    • 数据库兜底（最稳妥）：利用业务数据库作为最终一致性载体。在业务操作的事务中，除了业务数据，还在同一数据库的消息队列表里插入一条记录。由专门的Dispatcher组件负责从这张表拉取数据，发送到MQ；如果MQ失败，Dispatcher会根据重试次数不断重试。
    • 异构MQ双活：同时接入两个MQ集群（如RocketMQ + Kafka，或两个不同机房的RocketMQ）。发送消息时，采用双写（主写成功即返回，从异步写），消费时通过去重机制防止重复处理。当一个集群彻底宕机，流量自动切换到另一个。

3. 推荐架构：数据库事务消息表模式

这是最经典的“只靠MySQL也能顶住”的方案：

  1. 生产端：在同一个数据库事务中，完成业务操作（如更新订单状态）和向message_queue表插入一条状态为待发送的消息。
  2. 中间件：一个后台定时任务（Quartz或XXL-JOB），每秒扫描message_queue表中“待发送”且超过一定时间未确认的消息，将其发送至MQ。
  3. 消费端：消费业务时，先处理业务逻辑，再调用生产端的API（或直接更新数据库）将message_queue表中对应消息的状态更新为已发送或已完成。

4. 方案要点与取舍

在设计兜底方案时，有几个核心原则需要把握：

  • 最终一致性：MQ故障期间，系统承诺“最终一定好”，但不保证“立刻好”，以此换取系统的整体可用。
  • 幂等性：兜底方案往往涉及重复发送，因此消费端实现幂等是保障数据正确的基石。
  • 监控告警：无论是本地文件积压还是数据库队列表数据量过大，都必须配备监控，以便故障时人工介入。

总的来说，如果没有复杂的架构条件，建议优先实现“本地数据库消息表+定时任务”方案。如果追求极致的性能与可用性，可以考虑“双MQ写入+自动降级”方案。

核心解题思路：分组法

利用 ROW_NUMBER() 生成连续序列，通过计算日期与排名的差值来划分不同的连续区间。

第一步：数据去重

如果一个用户同一天登录多次，需要进行去重，确保每个用户每天只有一条记录。

第二步：生成排名（窗口函数）

使用 ROW_NUMBER() 为每个用户的登录日期按时间顺序编号（分区内连续且唯一）。

如果登录中断，login_date 的增加会跳过中断的日期，而 rn 依然连续增加，导致差值变化。

进阶变体：考虑用户可能跨月/跨年

date_sub(login_date, interval rn day) 这个方法已经兼容跨月跨年，因为 login_date 和 rn 都是数值，相减得到的是连续区间的不变标识。

  3. 这种方法通常需要结合 COUNT 来确保中间没有缺失，或者直接用 DATEDIFF 判断。

但总的来说，第一种 ROW_NUMBER + DATE_SUB 方法是目前公认最简洁、最通用的解法。

Java_GC停顿调优经验分享

1. 剖析GC停顿的根源

GC停顿的本质是JVM在进行内存回收时，为了确保数据一致性，必须暂停所有应用线程。在实际业务中，以下几种情况会显著延长STW时间或增加其发生频率：

  • 堆内存配置失衡：这是最常见的原因之一。例如，堆内存（-Xms与-Xmx）设置过小，会导致GC频繁触发；而新生代与老年代比例不当，则可能让短生命周期的对象过早晋升到老年代，引发代价高昂的Full GC。
  • GC算法与场景不匹配：不同的垃圾回收器有各自的适用场景。
    • Parallel GC 追求高吞吐量，但会在GC时产生较长的停顿，适合批处理任务。
    • G1 GC 通过将堆划分为多个Region，并设定停顿时间目标（-XX:MaxGCPauseMillis）来平衡吞吐量与延迟，但在并发回收失败或处理巨型对象（Humongous Object）时，也可能出现长时间的”转移阶段”STW。
    • ZGC/Shenandoah 则面向超大堆、超低延迟场景，通过着色指针和读屏障等先进技术，将STW时间几乎压缩到极致（<10ms），但可能会牺牲一些吞吐量或CPU资源。
  • 对象分配速率过高与内存泄漏：业务代码如果短时间内创建大量临时对象，会迅速填满Eden区，导致高频次的Minor GC。更严重的是内存泄漏，比如静态集合类未清理、缓存未设过期策略等，会导致老年代持续增长，最终引发频繁且耗时的Full GC。

2. 定位问题：诊断先行

面对GC问题，我一般不会盲目调整参数，而是先通过监控和日志来定位根因。

  • 开启详细GC日志：这是最基础也是最关键的一步。通过添加-XX:+PrintGCDetails -XX:+PrintGCDateStamps -Xloggc:/path/to/gc.log等参数，可以记录下每一次GC的耗时、前后内存变化以及触发原因。重点关注Minor GC频率（如每秒超过1次需警惕）和Full GC耗时（如超过500ms则需立即处理）。
  • 使用监控工具：利用jstat -gcutil <pid> 1s可以实时观察内存各区域使用率和GC计数。对于更深度的分析，我会使用 Eclipse MAT（Memory Analyzer Tool） 来分析堆转储文件（Heap Dump，通过jmap命令获取）。通过MAT的Dominator Tree（支配树）可以快速找出占用内存最大的对象，并追溯到它的GC Roots引用链，从而精准定位内存泄漏点。

3. 调优实战：分层优化策略

根据诊断结果，我会从以下几个层面进行针对性调优：

基础配置调优

首先确保基础设置合理。通常建议将-Xms和-Xmx设置为相同的值，避免JVM运行时动态调整堆大小带来的额外开销。然后根据应用特点选择回收器：

  • 平衡型Web服务：优先考虑 G1 GC（-XX:+UseG1GC），并设置一个可接受的暂停时间目标，如-XX:MaxGCPauseMillis=200。
  • 超低延迟服务：在JDK 11+环境下，可以考虑 ZGC（-XX:+UseZGC），追求亚毫秒级的停顿。

深度G1调优案例

在一个电商大促案例中，我们发现G1 GC的混合回收（Mixed GC）耗时超过1秒。通过分析GC日志和堆转储，发现老年代使用率持续超过80%，根源是Spring的ConcurrentMapCache缓存了大量未过期的数据。我们的解决方案是双管齐下：一是优化代码，将本地缓存实现替换为支持过期策略的Caffeine；二是调整参数，适当降低混合回收的触发阈值-XX:G1MixedGCLiveThresholdPercent，让G1更早、更平滑地处理老年代垃圾。

在另一个处理HBase海量数据的场景中，为了将GC暂停控制在100ms内，我们对G1进行了更精细的调优：

  • 启用并行引用处理 -XX:+ParallelRefProcEnabled，利用多线程处理软引用、弱引用，将标记阶段的耗时减少了75%。
  • 固定新生代占比 通过-XX:G1NewSizePercent将Eden区大小控制在1GB左右（对于100GB堆）。因为我们通过分析发现，G1清空Eden的速度大约是1GB/100ms。这样设置后，G1就能轻松满足设定的停顿时间目标。

新一代GC的应用

在一些对延迟极其敏感的服务中，我们引入了ZGC。ZGC的核心优势在于它通过着色指针（Colored Pointers） 和读屏障（Load Barrier） 技术，将标记、转移和重定位阶段都设计为并发执行。这意味着，原本在G1中耗时最长的、STW的”转移阶段”，在ZGC中几乎完全并发，其STW时间不随堆大小和活跃对象数量增加而增加，从而实现了<10ms的稳定停顿。

代码层面的治理

JVM参数调优是”治标”，代码质量才是”治本”。

  • 减少对象创建：避免在循环中创建大的临时对象，使用StringBuilder替代字符串拼接，复用对象或使用对象池。
  • 修复内存泄漏：对于静态集合类，可以考虑使用WeakHashMap；对于未关闭的资源（如文件流、数据库连接），务必使用try-with-resources语句确保及时释放。

总结

总的来说，应对GC引起的STW停顿，是一个从监控（发现问题）→ 分析（定位问题）→ 优化（解决问题） 的持续闭环。它既需要我们对JVM的内存模型和GC算法有深入理解，也要求我们具备良好的代码设计能力。我的经验是，优先从代码层面减少内存分配压力，再配合与业务场景相匹配的GC算法及参数调优，这样才能将GC对系统的影响控制在最低水平，确保服务的稳定和高效。
