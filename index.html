乐观锁实现高并发状态一致

在高并发场景下，多个线程或服务可能同时尝试修改同一条记录的“状态”字段，导致数据不一致（例如“状态”被错误覆盖或丢失更新）。乐观锁是一种轻量级的并发控制策略，它假设冲突较少，通过版本号或条件更新来保证数据一致性，而不使用数据库的行锁。

乐观锁的实现方式

1. 版本号机制

在数据库表中增加一个整数类型的版本号字段（如 version），每次读取数据时同时获取 version，更新时检查当前版本号是否与读取时的版本号一致。若一致则执行更新并将 version 加1；若不一致，说明数据已被其他事务修改，当前操作失败，需要重试或报错。

2. CAS（Compare And Swap）条件更新

直接利用数据库的原子性，在更新时通过 WHERE 条件携带旧的状态值，确保状态未被其他事务修改。这种方式不需要额外的版本字段，但要求状态字段在更新过程中具有业务上的不可重复性（即旧状态唯一确定当前记录）。

注意事项

  • ABA问题：CAS机制可能遇到ABA问题（值从A变成B又变回A），但状态字段如果业务上允许回环（如订单从“已支付”又回到“待支付”），版本号机制能更好地避免此问题。
  • 重试策略：高并发下冲突概率可能较高，需合理设置重试次数和退避策略（如指数退避），避免无限循环。
  • 最终一致性：乐观锁适用于冲突较少的场景，若冲突频繁，性能可能下降，此时应考虑悲观锁或其他分布式锁方案。

总结

乐观锁通过版本号或条件更新，在不锁定数据库行的情况下保证高并发下“状态”字段的一致性。实现简单，性能较好，但需要应用层处理重试逻辑，适合读多写少的业务场景。

Java线程池异常处理机制

在Java线程池中，当一个线程在执行任务时遇到异常（报错），这个异常会被任务本身的异常处理机制捕获（如果任务没有捕获异常，则会被抛出）。但是，线程池会如何处理这个线程呢？这个线程会阻塞还是会被复用于执行下一个任务？

关键点在于：线程池中的线程是会被复用的，即使一个任务执行过程中抛出了异常，该线程也不会因为异常而阻塞，而是会被线程池回收，并准备执行下一个任务。但是，异常的处理方式取决于任务本身和线程池的配置。

具体来说：

  1. 如果任务中抛出了异常并且没有被捕获（即抛出了未检查异常或检查异常，而任务没有处理），那么该线程会终止，并且异常会被传递到线程的未捕获异常处理器（如果设置了的话）。但是，在线程池中，这个线程可能会被新的线程替换（如果线程池允许的话）以维持线程池中的线程数。
  2. 实际上，对于Java的ThreadPoolExecutor，当一个线程因为任务抛出异常而结束时，线程池会创建一个新的线程来替换它（如果当前线程数小于最大线程数且线程池需要保持工作线程数）。这样，线程池中的线程数可能会动态调整。
  3. 但是，如果线程池中所有的线程都因为异常而结束，而线程池又没有创建新的线程（比如已经达到最大线程数且任务队列已满），那么线程池可能会进入一种状态，无法执行新提交的任务。
  4. 为了避免由于任务异常导致线程终止，我们通常会在任务内部捕获异常并进行处理，这样线程就可以继续执行下一个任务。
  5. 另外，可以通过实现Thread.UncaughtExceptionHandler来设置线程的默认未捕获异常处理器，但即使这样，线程在抛出未捕获异常后也会终止。在线程池中，终止的线程会被新的线程替换（如果线程池需要）。
  6. 还有一种方式是使用Future来获取任务执行结果，这样异常会被封装在Future中，可以通过Future.get()抛出异常，而线程池中的线程则不会因为异常而终止，因为异常已经被捕获并封装。

因此，总结来说：线程池中的线程遇到报错（未捕获异常）时，该线程会终止，但线程池可能会创建一个新的线程来替换它，以保持线程池中的线程数。所以，线程池不会因为一个线程的异常而阻塞，而是会继续运行，并尝试复用或创建新的线程来执行后续任务。

但是，如果任务中抛出了异常，并且我们希望在任务级别进行处理，而不影响线程的复用，那么最好在任务内部捕获异常。例如：

或者使用Callable任务，通过Future来获取异常。

注意：使用submit()方法提交任务，如果任务抛出异常，异常会被Future对象捕获，当调用Future.get()时，异常会被抛出。而使用execute()方法提交任务，异常会直接抛出到线程的未捕获异常处理器。

所以，为了线程的复用和稳定性，建议在任务内部处理异常，或者使用submit提交任务并通过Future处理异常。这样线程池中的线程就不会因为异常而终止，从而可以复用。

借证业务状态机建模原子性

1. 如何建模这些状态？

在借证业务中，我们会将流程拆解为状态、事件和动作三个核心维度。

核心状态定义

通常一个标准的借证流程可以抽象为以下几个关键状态（以枚举形式定义）：

  • 待申请：用户填写申请信息，但未正式提交。
  • 待审批：用户已提交，等待管理员/审批人审核。
  • 待出借：审批通过，等待管理员线下/线上发放证件（或生成电子证照）。
  • 使用中：证件已发放给借用人，处于借用期间。
  • 待归还：借用人发起归还申请（或临近归还期限）。
  • 待核验：管理员收到归还物品，进行完整性/一致性检查。
  • 已完成：流程正常结束，证件归档。
  • 已驳回/已拒绝：审批未通过。
  • 已作废/异常终止：流程因特殊原因终止。

业务流程流转建模

在实际建模时，我们不仅要关注状态，还要关注状态之间的合法转换路径。例如：

  • 待审批 -> 待出借 （审批通过）
  • 待审批 -> 已驳回 （审批拒绝）
  • 使用中 -> 待归还 （用户点击归还）
  • 待核验 -> 已完成 （核验无误，流程终结）

2. 有没有用到状态机设计模式？

是的，必须用到。 在借证这种业务逻辑复杂的场景中，引入状态机设计模式几乎是标准做法，它能有效避免 if/else 嵌套导致的逻辑混乱。

常用的实现方式：

1. 状态模式（State Pattern）

  • 思路：为每一个状态定义一个类，将状态对应的行为（如审批、归还）封装在该类中。
  • 适用场景：状态数量固定，但状态内的行为逻辑极复杂。不过，这种模式在借证业务中类会比较多，管理起来稍显繁琐。

2. 状态机引擎（推荐，如 Spring StateMachine 或自研）

  • 思路：通过配置文件、注解或 DSL（领域特定语言）来定义状态流转。

3. 如何确保状态转换的原子性？

这是分布式系统或高并发环境下最容易出问题的地方。假设两个人同时操作同一个借证流程（比如同时审批通过，或用户同时点击归还），必须保证数据的一致性。

通常采用以下几种策略组合来保证原子性：

1. 数据库乐观锁（推荐）

这是最常用的手段。在数据库表（如 borrow_record）中增加一个 version 字段。

2. 数据库行锁（悲观锁）

在事务中，使用 SELECT ... FOR UPDATE 锁定要修改的行，直到事务提交才释放。

3. 分布式锁（跨服务场景）

总结

在借证业务中：

  1. 建模：通过枚举定义清晰的状态和事件，绘制出合法的状态转换图。
  2. 设计模式：引入状态机引擎或采用状态模式，将复杂的流程流转从业务代码中抽离出来，提升代码的可维护性。
  3. 原子性：结合数据库乐观锁（基于版本号或状态条件更新）是核心手段，在高并发或跨服务场景下辅以分布式锁，确保状态不会错乱。

RAG流程详解与图解

RAG（Retrieval-Augmented Generation，检索增强生成）是一种结合信息检索与大语言模型的技术范式，旨在通过外部知识库增强生成内容的准确性和时效性。下面将从核心流程、关键组件以及多种图解视角，对RAG进行详细阐述。

一、RAG 核心流程概述

RAG 系统通常分为两个主要阶段：索引阶段（离线） 和 查询阶段（在线）。

  1. 索引阶段：将原始文档库分割、向量化并构建可快速检索的索引。
  2. 查询阶段：接收用户问题，检索相关文档片段，并将其与问题一同输入大模型生成答案。

二、详细流程步骤

1. 索引构建（离线）

  • 文档加载与解析：从多种格式（PDF、网页、数据库等）读取原始数据。
  • 文本分割（Chunking）：将长文档切分成适当大小的片段（chunk），通常考虑语义边界（如段落、句子）和 token 限制。
  • 向量化：使用 Embedding 模型将每个文本片段转换为稠密向量。
  • 存储索引：将向量及其对应的元数据（如原文、来源、位置）存入向量数据库（如 Pinecone、Weaviate、FAISS）。

2. 查询处理（在线）

  • 问题向量化：将用户问题用相同的 Embedding 模型转换为向量。
  • 相似性检索：在向量数据库中执行近似最近邻搜索（ANN），召回与问题最相似的 Top-K 个文本片段。
  • 结果融合与排序：可选地对检索结果进行重排序（Rerank），去除冗余，提高相关性。
  • 提示构造：将用户问题和检索到的文本片段按特定模板组装成提示词（Prompt），例如：
  • 生成：将提示词输入大语言模型（LLM），生成最终答案。
  • 输出与溯源：返回答案，并可附带引用来源（如原文片段）。

四、RAG 的变体与优化

  • HyDE（Hypothetical Document Embeddings）：先用 LLM 根据问题生成一个假想文档，再用该文档向量检索，提高召回率。
  • 递归检索：对检索结果再递归检索，获取更全面的上下文。
  • 多路召回：结合稀疏检索（如 BM25）和稠密检索，融合结果。
  • 自适应检索：根据问题复杂度决定是否检索或检索多少次。
  • 重排序：使用交叉编码器模型对初筛结果精细排序。

五、总结

RAG 流程通过“检索+生成”的协同工作，有效解决了大模型知识截止、幻觉等问题。其核心在于离线构建高质量的向量索引，在线执行高效的语义检索，并将检索结果作为上下文注入提示词，引导模型生成更准确的答案。上述多种图解从不同维度展示了这一流程，可根据实际需求选择或组合使用。

LLM参数原理解析

1. Temperature（温度系数）

作用： 控制概率分布的平滑程度（即高概率词和低概率词之间的差距）。

  • 原理：
    在应用 Softmax 函数之前，模型会输出每个词的分数（Logits，即未归一化的对数概率）。Temperature 参数 $T$ 会先对 Logits 进行缩放：
    $$
    \text{新 Logits} = \frac{\text{原始 Logits}}{T}
    $$
    然后再将缩放后的 Logits 输入 Softmax 得到最终的概率分布。
  • 效果：
    • $T < 1$（例如 0.5）： Logits 被放大（除以小于1的数）。这会使得原本高概率的词概率变得更高，低概率的词概率变得更低。结果就是模型更加保守、确定，倾向于选择最有可能的词。
    • $T = 1$： 保持原始分布不变。
    • $T > 1$（例如 1.5）： Logits 被缩小。概率分布变得更加平坦、均匀。原本低概率的词被选中的机会大大增加，结果就是模型更加随机、有创意，但也更容易跑题或产生语法错误。
    • $T \to \infty$： 所有词的概率趋近于相等，变成了完全随机选择。
    • $T \to 0$： 最高概率的词的概率趋近于 1，这实际上就变成了贪心搜索（Greedy Search）。

2. Top-K

作用： 一种截断策略，只从概率最高的 K 个词中挑选下一个词。

  • 原理：
    1. 模型计算出所有词的概率。
    2. 对概率进行排序，只保留概率排名前 K 个的词。
    3. 将这 K 个词的概率重新进行归一化（让它们的概率之和等于 1）。
    4. 从这 K 个词中进行随机采样（此时可以结合 Temperature 使用）。
  • 效果：
    • 固定值限制： 它强行排除了长尾的低概率词，防止模型说出特别不合时宜的话。
    • 缺点： K 是固定的，但不同语境下的概率分布形状是不同的。有时模型对前3个词都非常确信（分布很尖），Top-K=50 时就会把很多实际上不太可能的词也拉进来，增加了噪音。有时模型不太确定，前10个词的概率都差不多（分布很平），Top-K=10 可能会把一些原本合理的词（第11名）排除掉。

3. Top-P（核采样）

作用： 也是一种截断策略，但它不是固定数量截断，而是选择累积概率超过阈值 P 的最小集合。

  • 原理：
    1. 模型计算出所有词的概率。
    2. 按概率从高到低排序。
    3. 从最高的词开始累加概率，直到累加的概率和 超过 设定的阈值 P（例如 0.9）。
    4. 保留被累加的这些词（即累积概率达到 P 的核心词集合），舍弃其余的词。
    5. 将保留下来的词的概率重新归一化，然后进行采样。
  • 效果：
    • 动态调整： Top-P 比 Top-K 更灵活。它会根据当前预测的置信度动态调整候选词的数量。
    • 举例：
      • 如果模型非常确定，前3个词的累积概率已经达到了 0.95（P=0.9），那么它只保留这3个词。
      • 如果模型不太确定，需要保留 100 个词才能累积到 0.9 的概率，那么它就保留这 100 个词。

四者关系与协同工作

在实际的模型推理中，它们通常是组合使用的：

  1. 计算 Logits：模型前向传播得到原始分数。
  2. Temperature 缩放：先用 $T$ 调整 Logits，改变分布的锐度。
  3. 截断（Top-K 和/或 Top-P）：
    • 通常会先执行 Top-K（如果设置了且 $K > 0$），截断到 K 个词。
    • 然后再对剩下的词执行 Top-P，进一步动态筛选。
    • 或者只使用 Top-P。
  4. 重新归一化：对最终筛选出的词的概率进行 Softmax（或直接归一化）。
  5. 采样：从最终的候选集中随机抽取一个词作为输出。

总结

参数 控制方式 核心目的 对输出的影响
Temperature 分布形态 调整置信度 低值 -> 稳定、确定；高值 -> 多样、创造、随机
Top-K 数量硬截断 排除绝对的低频词 固定候选数，防止胡说，但缺乏语境适应性
Top-P 质量动态截断 保留主要概率质量 动态候选数，平衡质量和多样性，是目前的主流做法

通常建议，如果不确定怎么调，可以先固定 Temperature（如 0.8-1.0），然后主要调节 Top-P（如 0.9-0.95）。如果需要更确定性的回答，可以适当降低 Temperature 或同时降低 Top-P。

Executor_submit与execute方法区别

在Java的并发编程中，Executor框架提供了线程池的管理。其中，Executor接口只有一个execute方法，而ExecutorService接口（继承自Executor）增加了submit方法。主要区别如下：

  1. 返回值：
    • execute方法没有返回值，它的定义是void execute(Runnable command)。因此，如果你不需要任务执行的结果，可以使用它。
    • submit方法返回一个Future对象，可以用来获取任务执行的结果或取消任务。它有多个重载版本，可以接受Runnable或Callable任务。
  2. 异常处理：
    • 使用execute方法时，任务中的异常必须在任务内部处理（例如通过try-catch），否则异常会传递到线程池的未捕获异常处理器（如果设置了的话），但不会通过execute方法抛出。
    • 使用submit方法时，任务中的异常可以通过返回的Future对象的get方法抛出。当调用Future.get()时，如果任务中抛出了异常，那么get方法会抛出ExecutionException，并将原始异常作为原因。
  3. 任务类型：
    • execute只接受Runnable任务。
    • submit可以接受Runnable和Callable任务。Callable任务可以返回结果和抛出异常，而Runnable任务不能。
  4. 使用场景：
    • 如果你只是想要异步执行一个任务，并且不关心结果，那么使用execute。
    • 如果你需要任务执行的结果，或者需要处理任务中可能抛出的异常，那么使用submit。

Java_Synchronized底层实现机制详解

Java中的synchronized关键字是用于实现线程同步的，它能够确保在同一时刻最多只有一个线程可以执行某个方法或代码块。synchronized可以用于方法（实例方法和静态方法）以及代码块。

底层工作原理：

  1. synchronized的实现依赖于JVM层面的监视器锁（Monitor），每个Java对象都可以作为一个监视器锁。当线程进入synchronized方法或代码块时，它会自动获取对象的监视器锁，退出时（无论是正常退出还是异常退出）会自动释放锁。
  2. 在JVM中，监视器锁是通过对象的对象头中的Mark Word来实现的。Mark Word中有一个字段指向一个称为监视器（Monitor）的指针，这个监视器实际上是一个C++实现的对象（ObjectMonitor），它维护了多个队列来管理线程的等待和唤醒。
  3. 当一个线程尝试获取对象的监视器锁时，如果锁没有被其他线程占用，那么该线程就会成为该锁的拥有者，并且将锁的计数器加1。如果该锁已经被其他线程占用，那么该线程就会进入同步队列（SynchronizedQueue）中，并进入阻塞状态，直到锁被释放。
  4. 当线程释放锁时，它会将锁的计数器减1，如果计数器为0，则完全释放锁，然后会从同步队列中唤醒一个线程（具体唤醒哪一个取决于实现，可能是公平的，也可能是非公平的）。
  5. 在早期的Java版本中，synchronized是重量级锁，需要从用户态切换到内核态，性能开销较大。但是经过后续的优化，引入了偏向锁、轻量级锁等机制，使得synchronized的性能有了很大提升。
    • 偏向锁：在无竞争的情况下，整个同步过程就消除了同步开销。当一个线程访问同步代码块时，会在对象头和栈帧中的锁记录里存储锁偏向的线程ID，以后该线程进入和退出同步块时不需要进行CAS操作来加锁和解锁。
    • 轻量级锁：当存在锁竞争但竞争不激烈时，线程不会阻塞，而是通过自旋（CAS操作）来尝试获取锁，从而避免了用户态到内核态的切换。
    • 重量级锁：当竞争激烈时，轻量级锁会升级为重量级锁，此时未获取到锁的线程会进入阻塞状态，需要操作系统进行线程调度和上下文切换。
  6. 锁的升级过程：无锁 -> 偏向锁 -> 轻量级锁 -> 重量级锁。这个过程是为了在无竞争或低竞争的情况下减少锁的开销。
  7. synchronized的底层实现还涉及到内存屏障（Memory Barrier）来保证可见性和有序性。在进入synchronized块时，会执行一个acquire操作，该操作会刷新处理器缓存，使得当前线程可以看到之前线程对共享变量的修改。在退出synchronized块时，会执行一个release操作，将当前线程对共享变量的修改刷新到主内存中，保证其他线程可见。

总结：synchronized通过JVM内置的Monitor实现，结合了偏向锁、轻量级锁和重量级锁等多种锁机制，以及内存屏障，来保证线程安全、可见性和有序性。这些优化使得synchronized在大多数情况下都能有较好的性能表现。

三、锁的升级优化（JDK 1.6+）

为了平衡性能，JVM采用了锁升级策略：

1. 无锁状态

  • 对象刚创建时的初始状态

2. 偏向锁（Biased Locking）

  • 适用场景：只有一个线程访问
  • 原理：记录线程ID，同一线程重入时无需CAS操作
  • 优势：几乎没有同步开销

3. 轻量级锁（Lightweight Locking）

  • 适用场景：多个线程交替执行，没有实际竞争
  • 原理：通过CAS将对象头替换为指向线程栈中锁记录的指针
  • 失败时膨胀为重量级锁

4. 重量级锁（Heavyweight Locking）

  • 适用场景：多线程激烈竞争
  • 原理：依赖操作系统的互斥量（Mutex）实现
  • 特点：线程阻塞和唤醒需要内核态切换，开销较大

四、内存语义

synchronized保证三个特性：

  1. 原子性：通过互斥访问保证
  2. 可见性：通过以下内存屏障实现：
    • 进入同步块：Load Barrier（读取主内存）
    • 退出同步块：Store Barrier（写入主内存）
  3. 有序性：禁止指令重排序（as-if-serial语义）

  4. 锁消除：JVM会对不可能存在竞争的锁进行消除
  5. 锁粗化：JVM会将连续的同步块合并

总结

synchronized的底层实现是一个从无锁→偏向锁→轻量级锁→重量级锁的渐进式升级过程。这种设计在保证线程安全的同时，尽可能减少了同步开销，是Java并发编程的基石之一。
