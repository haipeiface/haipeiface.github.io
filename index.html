一

Disruptor的简介

Disruptor是基于事件异步驱动模型实现的，采用了RingBuffer数据结构，支持高并发、低延时、高吞吐量的高性能工作队列，它是由英国外汇交易公司LMAX开发的，研发的初衷是解决内存队列的延迟问题，不同于我们常用的分布式消息中间件RocketMQ、Kafaka，而Disruptor是单机的、本地内存队列，类似JDK的ArrayBlockingQueue等队列。

Disruptor的使用场景

  • 加密货币交易撮合引擎
  • Log4j2基于Disruptor实现的异步日志处理
  • Canal+Disruptor实现高效的数据同步
  • 知名开源框架Apache Strom

2010年在QCon的演讲，介绍了基于Disruptor开发的系统单线程能支撑每秒600万订单，由此可见该组件可以大幅提升系统的TPS，所以对于一些需要大幅提升单机应用的吞吐量的场景可以考虑使用Disruptor。

Disruptor和ArrayBlockingQueue性能对比

  • ArrayBlockingQueue是基于数组ArrayList实现的，通过ReentrantLock独占锁保证线程安全；
  • Disruptor是基于环形数组队列RingBuffer实现的，通过CAS乐观锁保证线程安全。在多种生产者-消费者模式下的性能对比。

二

Disruptor的核心概念

Disruptor内部组件交互图

核心概念

有些概念前面已经介绍过，在此不再赘述，说一说还未介绍的几个概念：

Sequence

Sequence本身就是一个序号管理器，它是严格顺序增长的，Disruptor通过它标识和定位RingBuffer中的每一个事件，每个Consumer都维护一个Sequence，通过Sequence可以跟踪Consumer事件处理进度，它有AtomicLong的大多数功能特性，而且它消除了CPU伪共享的问题。

Sequencer

Sequencer是一个接口，它有两个实现类：SingleProducerSequencer(单生产者实现)、MultiProducerSequencer(多生产者实现)，它主要作用是实现生产者和消费者之间快速、正确传递数据的并发算法。

Sequencer是生产者与缓冲区RingBuffer之间的桥梁。生产者可以通过Sequencer向RingBuffer申请数据的存放空间，并使用publish()方法通过WaiteStrategy通知消费者。

SequenceBarrier（序列屏障）

SequenceBarrier用于保证事件的有序性。它通过维护一组Sequence来跟踪消费者的进度，当生产者发布新的事件时，序列屏障会检查是否所有消费者都已处理完前面的事件，如果是，则通知生产者可以发布新的事件。

SequenceBarrier是消费者与RingBuffer之间的桥梁。在Disruptor中，消费者直接访问的是SequenceBarrier，而不是RingBuffer，因此SequenceBarrier能减少RingBuffer上的并发冲突，当消费者的消费速度大于生产者的生产速度时，消费者就可以通过waitFor()方法给予生产者一定的缓冲时间，从而协调了生产者和消费者的速度问题。

SequenceBarrier同时也是消费者与消费者之间消费依赖的抽象，SequenceBarrier只有一个实现类，即ProcessingSequenceBarrier。ProcessingSequenceBarrier由生产者Sequencer、消费定位cursorSequence、等待策略waitStrategy、还有一组依赖Sequence(dependentSequence)组成。

三

Disruptor的特点

环形数组结构

  • 采用首尾相接的数组而非链表，无需担心index溢出问题，且数组对处理器的缓存机制更加友好；
  • 在RingBuffer数组长度设置为2^N时，通过sequence & (bufferSize-1)加速定位元素实际下标索引，通过结合左移(<<)操作实现乘法；
  • 结合SequenceBarrier机制，实现线程与线程之间高效的数据交互。

无锁化设计

每个生产者或者消费者线程，会先申请可以操作的元素在数组中的位置，申请到之后，直接在该位置写入或者读取数据，整个过程通过原子变量CAS，保证操作的线程安全，即Disruptor的Sequence的自增就是CAS的自旋自增，对应的ArrayBlockQueue的数组索引index是互斥自增。

独占缓存行的方式消除伪共享

什么是伪共享

出现伪共享问题（False Sharing）的原因：

  • 一个缓存行可以存储多个变量（存满当前缓存行的字节数）；64个字节可以放8个long，16个int；
  • 而CPU对缓存的修改又是以缓存行为最小单位的；不是以long 、byte这样的数据类型为单位的；
  • 在多线程情况下，如果需要修改“共享同一个缓存行的其中一个变量”，该行中其他变量的状态就会失效，甚至进行一致性保护。

所以，伪共享问题（False Sharing）的本质是：

CPU针对缓存的操作是以Cache Line为基本单位，对缓存行中的单个变量进行修改，会导致整个缓存行其他不相关的数据也都失效了，需要从主存重新加载，这个过程会带来性能损耗。

Disruptor是如何解决伪共享的

Sequence是标识RingBuffer环形数组的下标，同时生产者和消费者也会维护各自的Sequence，最重要的是，Sequence通过填充CPU缓存行避免了伪共享带来的性能损耗，来看下其填充缓存行源码：

撮合应用的特点

  • 纯内存的、CPU密集型的

应用启动时加载数据库未处理订单、写日志、撮合成功发送消息到MQ会涉及IO操作。

  • 有状态的

正因为应用是有状态的，所以需要通过Disruptor提升单机的性能和吞吐量。

为什么撮合应用不设计成无状态的？

在学习或者实际做架构设计时，一般大多数情况都建议将应用设计为无状态的，可以通过水平扩展，实现应用的高可用、高性能。而有状态的应用一般有单点故障问题，难以通过水平扩展提升应用的性能，但是做架构设计的时候，还是需要从实际的场景出发，而撮合应用场景很显然更适合设计成有状态的。在数字加密货币交易平台，每一种数字加密货币都是由唯一的“交易对”去标识的，类似股票交易中的股票代码，针对不同交易对的买卖交易单是天然隔离的，而同种交易对的买卖交易单必须是在同一个应用去处理的，否则匹配撮合的时候是有问题的。如果使用无状态的设计，那么所有的交易对都必须在一个集群内处理，而且每个应用都必须要有全量交易对的订单数据，这样就会存在两个问题：多个应用撮合匹配结果不一致，以哪个为准、热点交易对如何做隔离，所以解决方案就是根据交易对维度对订单做分片，同一个交易对的订单消息路由到同一个撮合应用进行处理，这样其实就是将撮合应用设计成有状态的。每一种交易对每个时刻有且只有一个应用能处理，然后再通过k8s的Liveness和Readiness探针做自动故障转移和恢复来解决单点故障的问题，最后通过本地缓存Caffeine+高性能队列Disruptor提升单pod的吞吐量。16C64G的配置在实际业务场景压测的结果是，单机最大TPS在200w/s左右，对于整个交易系统而言性能瓶颈已经不在撮合应用，因为极端情况下可以配置成一个pod处理一个交易对。

乐观锁实现高并发状态一致

在高并发场景下，多个线程或服务可能同时尝试修改同一条记录的“状态”字段，导致数据不一致（例如“状态”被错误覆盖或丢失更新）。乐观锁是一种轻量级的并发控制策略，它假设冲突较少，通过版本号或条件更新来保证数据一致性，而不使用数据库的行锁。

乐观锁的实现方式

1. 版本号机制

在数据库表中增加一个整数类型的版本号字段（如 version），每次读取数据时同时获取 version，更新时检查当前版本号是否与读取时的版本号一致。若一致则执行更新并将 version 加1；若不一致，说明数据已被其他事务修改，当前操作失败，需要重试或报错。

SQL示例：

如果更新影响的行数为0，说明版本号已被其他事务改变，此时应重新查询最新状态和版本号，并再次尝试更新（重试机制）。

2. CAS（Compare And Swap）条件更新

直接利用数据库的原子性，在更新时通过 WHERE 条件携带旧的状态值，确保状态未被其他事务修改。这种方式不需要额外的版本字段，但要求状态字段在更新过程中具有业务上的不可重复性（即旧状态唯一确定当前记录）。

注意事项

  • ABA问题：CAS机制可能遇到ABA问题（值从A变成B又变回A），但状态字段如果业务上允许回环（如订单从“已支付”又回到“待支付”），版本号机制能更好地避免此问题。
  • 重试策略：高并发下冲突概率可能较高，需合理设置重试次数和退避策略（如指数退避），避免无限循环。
  • 最终一致性：乐观锁适用于冲突较少的场景，若冲突频繁，性能可能下降，此时应考虑悲观锁或其他分布式锁方案。

总结

乐观锁通过版本号或条件更新，在不锁定数据库行的情况下保证高并发下“状态”字段的一致性。实现简单，性能较好，但需要应用层处理重试逻辑，适合读多写少的业务场景。

大模型输出标准JSON解决方法

思考过程（Thinking）：

我们被问到：“如何解决大模型不按要求输出标准json格式的问题”。这是一个关于大语言模型（LLM）应用开发中的常见问题：当要求模型输出JSON时，它可能输出带有额外解释、格式错误、或者不完全符合JSON规范的内容。

用户想要知道如何解决这个问题。作为助手，需要提供一些实用的方法和技术来强制或引导模型输出有效的JSON。

需要覆盖：

  1. 为什么模型会输出非标准JSON（自然语言习惯、缺少约束等）。
  2. 提示工程技巧：清晰指定格式、使用few-shot示例、明确要求“只输出JSON，不要其他内容”。
  3. 使用约束解码或结构化生成技术（如JSON模式、语法约束、使用函数调用/工具使用）。
  4. 后处理修复：解析失败时进行修复（如使用正则提取、json修复库如json5、demjson3）。
  5. 使用特定的API参数：如OpenAI的response_format参数（json_object或json_schema）。
  6. 其他方法：调整温度参数降低随机性，多次尝试并验证。
  7. 总结建议：最好结合提示工程和API参数，后处理作为兜底。

回答应结构化，提供步骤和代码示例（可能），但不要过于冗长。使用中文。

现在组织回答。

大模型不按要求输出标准 JSON 是一个常见问题，主要原因在于模型训练数据中包含大量自然语言和自由文本，对格式的约束力较弱。解决这一问题可以从提示工程、API 参数控制、结构化生成以及后处理修复四个层面入手。

1. 优化提示词（Prompt Engineering）

  • 明确指定输出格式：在指令中强调“只输出 JSON，不要包含任何解释、注释或 Markdown 代码块标记”。
  • 使用 Few-shot 示例：给出几个输入-输出的例子，让模型模仿格式。
  • 约束键和值类型：描述 JSON 的结构，例如“必须是一个包含 name（字符串）和 age（整数）的 JSON 对象”。

  • JSON 修复库：如 json5（允许注释、尾随逗号）、demjson3 或 json_repair（专门修复常见错误）。

Java线程池异常处理机制

在Java线程池中，当一个线程在执行任务时遇到异常（报错），这个异常会被任务本身的异常处理机制捕获（如果任务没有捕获异常，则会被抛出）。但是，线程池会如何处理这个线程呢？这个线程会阻塞还是会被复用于执行下一个任务？

关键点在于：线程池中的线程是会被复用的，即使一个任务执行过程中抛出了异常，该线程也不会因为异常而阻塞，而是会被线程池回收，并准备执行下一个任务。但是，异常的处理方式取决于任务本身和线程池的配置。

具体来说：

  1. 如果任务中抛出了异常并且没有被捕获（即抛出了未检查异常或检查异常，而任务没有处理），那么该线程会终止，并且异常会被传递到线程的未捕获异常处理器（如果设置了的话）。但是，在线程池中，这个线程可能会被新的线程替换（如果线程池允许的话）以维持线程池中的线程数。
  2. 实际上，对于Java的ThreadPoolExecutor，当一个线程因为任务抛出异常而结束时，线程池会创建一个新的线程来替换它（如果当前线程数小于最大线程数且线程池需要保持工作线程数）。这样，线程池中的线程数可能会动态调整。
  3. 但是，如果线程池中所有的线程都因为异常而结束，而线程池又没有创建新的线程（比如已经达到最大线程数且任务队列已满），那么线程池可能会进入一种状态，无法执行新提交的任务。
  4. 为了避免由于任务异常导致线程终止，我们通常会在任务内部捕获异常并进行处理，这样线程就可以继续执行下一个任务。
  5. 另外，可以通过实现Thread.UncaughtExceptionHandler来设置线程的默认未捕获异常处理器，但即使这样，线程在抛出未捕获异常后也会终止。在线程池中，终止的线程会被新的线程替换（如果线程池需要）。
  6. 还有一种方式是使用Future来获取任务执行结果，这样异常会被封装在Future中，可以通过Future.get()抛出异常，而线程池中的线程则不会因为异常而终止，因为异常已经被捕获并封装。

因此，总结来说：线程池中的线程遇到报错（未捕获异常）时，该线程会终止，但线程池可能会创建一个新的线程来替换它，以保持线程池中的线程数。所以，线程池不会因为一个线程的异常而阻塞，而是会继续运行，并尝试复用或创建新的线程来执行后续任务。

但是，如果任务中抛出了异常，并且我们希望在任务级别进行处理，而不影响线程的复用，那么最好在任务内部捕获异常。例如：

或者使用Callable任务，通过Future来获取异常。

注意：使用submit()方法提交任务，如果任务抛出异常，异常会被Future对象捕获，当调用Future.get()时，异常会被抛出。而使用execute()方法提交任务，异常会直接抛出到线程的未捕获异常处理器。

所以，为了线程的复用和稳定性，建议在任务内部处理异常，或者使用submit提交任务并通过Future处理异常。这样线程池中的线程就不会因为异常而终止，从而可以复用。

在Java线程池中，当线程执行任务遇到未捕获的异常时，线程不会阻塞，而是会被销毁，然后线程池会创建新的线程来替代它，以保持线程池的线程数量。

借证业务状态机建模原子性

1. 如何建模这些状态？

在借证业务中，我们会将流程拆解为状态、事件和动作三个核心维度。

核心状态定义

通常一个标准的借证流程可以抽象为以下几个关键状态（以枚举形式定义）：

  • 待申请：用户填写申请信息，但未正式提交。
  • 待审批：用户已提交，等待管理员/审批人审核。
  • 待出借：审批通过，等待管理员线下/线上发放证件（或生成电子证照）。
  • 使用中：证件已发放给借用人，处于借用期间。
  • 待归还：借用人发起归还申请（或临近归还期限）。
  • 待核验：管理员收到归还物品，进行完整性/一致性检查。
  • 已完成：流程正常结束，证件归档。
  • 已驳回/已拒绝：审批未通过。
  • 已作废/异常终止：流程因特殊原因终止。

业务流程流转建模

在实际建模时，我们不仅要关注状态，还要关注状态之间的合法转换路径。例如：

  • 待审批 -> 待出借 （审批通过）
  • 待审批 -> 已驳回 （审批拒绝）
  • 使用中 -> 待归还 （用户点击归还）
  • 待核验 -> 已完成 （核验无误，流程终结）

2. 有没有用到状态机设计模式？

是的，必须用到。 在借证这种业务逻辑复杂的场景中，引入状态机设计模式几乎是标准做法，它能有效避免 if/else 嵌套导致的逻辑混乱。

常用的实现方式：

1. 状态模式（State Pattern）

  • 思路：为每一个状态定义一个类，将状态对应的行为（如审批、归还）封装在该类中。
  • 适用场景：状态数量固定，但状态内的行为逻辑极复杂。不过，这种模式在借证业务中类会比较多，管理起来稍显繁琐。

2. 状态机引擎（推荐，如 Spring StateMachine 或自研）

  • 思路：通过配置文件、注解或 DSL（领域特定语言）来定义状态流转。

3. 如何确保状态转换的原子性？

这是分布式系统或高并发环境下最容易出问题的地方。假设两个人同时操作同一个借证流程（比如同时审批通过，或用户同时点击归还），必须保证数据的一致性。

通常采用以下几种策略组合来保证原子性：

1. 数据库乐观锁（推荐）

这是最常用的手段。在数据库表（如 borrow_record）中增加一个 version 字段。

2. 数据库行锁（悲观锁）

在事务中，使用 SELECT ... FOR UPDATE 锁定要修改的行，直到事务提交才释放。

  • 适用场景：并发冲突极高的场景，或者对一致性要求极其严格（如财务核心流程）。但需要注意锁等待和死锁风险，性能开销也较大。

3. 分布式锁（跨服务场景）

如果借证业务被拆分为多个微服务（比如审批服务、库存服务），状态变更涉及跨服务调用时，需要引入 Redis 或 Zookeeper 分布式锁。

  • 做法：以借证记录 ID 为 Key 加锁，确保只有一个线程能执行整个状态流转逻辑。

4. 状态机引擎的内部保证

好的状态机框架（如 Spring StateMachine）本身会结合上述机制，或者在内存中保证同一个状态机实例的串行执行。但在实际业务代码中，我们仍需在数据库层面做好最后的兜底校验。

总结

在借证业务中：

  1. 建模：通过枚举定义清晰的状态和事件，绘制出合法的状态转换图。
  2. 设计模式：引入状态机引擎或采用状态模式，将复杂的流程流转从业务代码中抽离出来，提升代码的可维护性。
  3. 原子性：结合数据库乐观锁（基于版本号或状态条件更新）是核心手段，在高并发或跨服务场景下辅以分布式锁，确保状态不会错乱。

RAG流程详解与图解

RAG（Retrieval-Augmented Generation，检索增强生成）是一种结合信息检索与大语言模型的技术范式，旨在通过外部知识库增强生成内容的准确性和时效性。下面将从核心流程、关键组件以及多种图解视角，对RAG进行详细阐述。

一、RAG 核心流程概述

RAG 系统通常分为两个主要阶段：索引阶段（离线） 和 查询阶段（在线）。

  1. 索引阶段：将原始文档库分割、向量化并构建可快速检索的索引。
  2. 查询阶段：接收用户问题，检索相关文档片段，并将其与问题一同输入大模型生成答案。

二、详细流程步骤

1. 索引构建（离线）

  • 文档加载与解析：从多种格式（PDF、网页、数据库等）读取原始数据。
  • 文本分割（Chunking）：将长文档切分成适当大小的片段（chunk），通常考虑语义边界（如段落、句子）和 token 限制。
  • 向量化：使用 Embedding 模型将每个文本片段转换为稠密向量。
  • 存储索引：将向量及其对应的元数据（如原文、来源、位置）存入向量数据库（如 Pinecone、Weaviate、FAISS）。

2. 查询处理（在线）

  • 问题向量化：将用户问题用相同的 Embedding 模型转换为向量。
  • 相似性检索：在向量数据库中执行近似最近邻搜索（ANN），召回与问题最相似的 Top-K 个文本片段。
  • 结果融合与排序：可选地对检索结果进行重排序（Rerank），去除冗余，提高相关性。

  • 生成：将提示词输入大语言模型（LLM），生成最终答案。
  • 输出与溯源：返回答案，并可附带引用来源（如原文片段）。

四、RAG 的变体与优化

  • HyDE（Hypothetical Document Embeddings）：先用 LLM 根据问题生成一个假想文档，再用该文档向量检索，提高召回率。
  • 递归检索：对检索结果再递归检索，获取更全面的上下文。

LLM参数原理解析

1. Temperature（温度系数）

作用： 控制概率分布的平滑程度（即高概率词和低概率词之间的差距）。

  • 原理：
    在应用 Softmax 函数之前，模型会输出每个词的分数（Logits，即未归一化的对数概率）。Temperature 参数 $T$ 会先对 Logits 进行缩放：
    $$
    \text{新 Logits} = \frac{\text{原始 Logits}}{T}
    $$
    然后再将缩放后的 Logits 输入 Softmax 得到最终的概率分布。
  • 效果：
    • $T < 1$（例如 0.5）： Logits 被放大（除以小于1的数）。这会使得原本高概率的词概率变得更高，低概率的词概率变得更低。结果就是模型更加保守、确定，倾向于选择最有可能的词。
    • $T = 1$： 保持原始分布不变。
    • $T > 1$（例如 1.5）： Logits 被缩小。概率分布变得更加平坦、均匀。原本低概率的词被选中的机会大大增加，结果就是模型更加随机、有创意，但也更容易跑题或产生语法错误。
    • $T \to \infty$： 所有词的概率趋近于相等，变成了完全随机选择。
    • $T \to 0$： 最高概率的词的概率趋近于 1，这实际上就变成了贪心搜索（Greedy Search）。

2. Top-K

作用： 一种截断策略，只从概率最高的 K 个词中挑选下一个词。

  • 原理：
    1. 模型计算出所有词的概率。
    2. 对概率进行排序，只保留概率排名前 K 个的词。
    3. 将这 K 个词的概率重新进行归一化（让它们的概率之和等于 1）。
    4. 从这 K 个词中进行随机采样（此时可以结合 Temperature 使用）。
  • 效果：
    • 固定值限制： 它强行排除了长尾的低概率词，防止模型说出特别不合时宜的话。
    • 缺点： K 是固定的，但不同语境下的概率分布形状是不同的。有时模型对前3个词都非常确信（分布很尖），Top-K=50 时就会把很多实际上不太可能的词也拉进来，增加了噪音。有时模型不太确定，前10个词的概率都差不多（分布很平），Top-K=10 可能会把一些原本合理的词（第11名）排除掉。

3. Top-P（核采样）

作用： 也是一种截断策略，但它不是固定数量截断，而是选择累积概率超过阈值 P 的最小集合。

  • 原理：
    1. 模型计算出所有词的概率。
    2. 按概率从高到低排序。
    3. 从最高的词开始累加概率，直到累加的概率和 超过 设定的阈值 P（例如 0.9）。
    4. 保留被累加的这些词（即累积概率达到 P 的核心词集合），舍弃其余的词。
    5. 将保留下来的词的概率重新归一化，然后进行采样。
  • 效果：
    • 动态调整： Top-P 比 Top-K 更灵活。它会根据当前预测的置信度动态调整候选词的数量。
    • 举例：
      • 如果模型非常确定，前3个词的累积概率已经达到了 0.95（P=0.9），那么它只保留这3个词。
      • 如果模型不太确定，需要保留 100 个词才能累积到 0.9 的概率，那么它就保留这 100 个词。

四者关系与协同工作

在实际的模型推理中，它们通常是组合使用的：

  1. 计算 Logits：模型前向传播得到原始分数。
  2. Temperature 缩放：先用 $T$ 调整 Logits，改变分布的锐度。
  3. 截断（Top-K 和/或 Top-P）：
    • 通常会先执行 Top-K（如果设置了且 $K > 0$），截断到 K 个词。
    • 然后再对剩下的词执行 Top-P，进一步动态筛选。
    • 或者只使用 Top-P。
  4. 重新归一化：对最终筛选出的词的概率进行 Softmax（或直接归一化）。
  5. 采样：从最终的候选集中随机抽取一个词作为输出。

总结

参数 控制方式 核心目的 对输出的影响
Temperature 分布形态 调整置信度 低值 -> 稳定、确定；高值 -> 多样、创造、随机
Top-K 数量硬截断 排除绝对的低频词 固定候选数，防止胡说，但缺乏语境适应性
Top-P 质量动态截断 保留主要概率质量 动态候选数，平衡质量和多样性，是目前的主流做法

通常建议，如果不确定怎么调，可以先固定 Temperature（如 0.8-1.0），然后主要调节 Top-P（如 0.9-0.95）。如果需要更确定性的回答，可以适当降低 Temperature 或同时降低 Top-P。

AI智能体类型介绍

1. 规划型智能体

  • Plan & Execute：将任务分解为“规划”和“执行”两步。先制定整体步骤，再逐一执行，比ReAct边想边做的模式更稳定，适合处理复杂流程。
  • 链式思维：通过“让我们一步一步思考”引导模型逐步推理，虽不直接调用工具，但能显著提升逻辑性，常作为其他智能体内部推理的模板。

2. 工具型智能体

  • Function Calling：模型根据用户指令，输出一个结构化的函数调用指令，由外部系统执行。特点是意图识别精准、返回结果结构化，非常适合集成到现有API系统中。
  • Toolformer / Gorilla：通过微调让模型学会在文本中插入“调用API”的特殊标记，能更灵活地决定何时使用工具。

3. 反思型智能体

  • Self-ask / Self-check：在执行中不断向自己提问（如“完成这个我需要知道什么？”），通过自问自答来检查逻辑，避免在错误方向越走越远。
  • Reflexion：一种强化学习模式。智能体尝试执行任务，将失败经验总结成文本存入记忆，下次执行时用来避免重复错误。

4. 多智能体协作

  • 多智能体辩论：让多个智能体扮演不同角色（如程序员、测试员）进行讨论和互相审核，以产出更可靠的结果。
  • AutoGen：微软框架，支持多个对话智能体协作。一个写代码，另一个负责执行和调试，像模拟一个微型团队。

5. 记忆型智能体

  • MemGPT：通过虚拟内存管理技术，让智能体在长时对话或处理大数据时，能主动在“主记忆”和“外部存储”之间交换信息。

6. 数据探索型

  • Data Agent：专门用于数据领域，能主动探索数据集的统计信息、尝试可视化，发现异常后再调整策略，而不是一次性生成最终图表。

7. 面向特定场景的范式

  • Self-consistency：对同一个问题生成多个推理路径，然后投票选出最一致的答案，多用于复杂数学或逻辑题。
  • Self-ask：通过结构化的搜索追问，特别适合需要频繁查找外部知识库的场景。

简单总结：ReAct适合快速执行简单任务；复杂任务可选Plan & Execute；需要高精度API交互可选Function Calling；高难度任务可尝试多智能体协作。

流程说明：

      • 步数加1，继续循环。
    • 如果无法按预期格式解析，则将错误信息加入历史，步数加1，继续循环。
